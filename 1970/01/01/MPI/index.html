MPI分布式内存编程

Message-Passing Interface

并不是一种语言，而是函数库

分布式内存

核-内存对由网络连接

与核相关联的内存只能由该核访问

用消息传递对分布式系统进行编程

一个进程调用发送函数，另一个进程调用接受函数

例程

setting： 核数：4

发送顺序是不确定的

接受是按顺序—-因为循环了设置了接受来源—-打印顺序是确定的（q）
MPI_Recv(greeting, MAX_STRING, MPI_CHAR, q, 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE);

但是后面可能已经完成了，要等到前面的发送(阻塞)

每个进程都发给0号进程，0号进程只是打印

MPI_ANY_SOURCE

可以按照接受顺序打印

MPI_Init 和 MPI_Finalize

MPI_Init： 告知MPI做所有必要的初始化

MPI_Finalize： 告知使用完毕，分配的资源可以释放

  • 一组可以互相发送消息的进程集合，指定通信范围
  • MPI_Init : 用户启动程序时，定义由用户启动的所有进程组成的通信子
    MPI_COMM_WORLD

返回通信子进程数(传指针进去修改)

MPI_Comm_rank()

返回进程号(传指针进去修改)

  • 一个程序，不同进程不同动作，

  • If-else 语句实现

MPI数据类型

接收方

  • dest ： 接受消息进程号

  • tag：非负int，区分看上去完全一样的消息

         比如，有些接受的数据打印，有的不需要，则用tag来标记

  • communicator ：通信子，指定通信范围

不同通信子不能通信

MPI_Recv

前六个与Send 对应

source： 匹配进程号，也可以MPI_ANY_SOURCE

按发送的顺序接受（就是不指定source，先到的先发）

tag 匹配
communicator匹配
status_p 大部分情况下不调用, 赋予MPI_STATUS_IGBORE

Matching

若q号发给r

  • recv_comm = send_comm
  • recv_tag = send_tag
  • dest = r
  • src = q

recv_type = send_type

recv_buf_sz >= send_buf_sz

接受顺序/阻塞

如果按照进程号接受结果，那么跑的快的必须等待跑的慢的

Send 与buffersize cutoffs blocking 有关

  • Recv总是阻塞，直到接受到

问题：悬挂，如果调用MPI_Send()发生了阻塞，没有相匹配的接受，发生进程被悬挂

MPI_ANY_SOURCE ,可以按接受顺序执行

MPI_Recv(greeting, MAX_STRING, MPI_CHAR, MPI_ANY_SOURCE , 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE);

那么对应的tag也可以是MPI_ANT_TAG

  • 发送可以阻塞或者缓冲发送的消息
    较小的消息会缓冲,较大的消息会选择阻塞模式(就是等待接手之后再发)
    如果每个进程都发可能会都在等待接收,造成死锁
    • 解决①,偶数先发再收,奇数先收再发
    • 解决②,用SendRecv( ),系统会调度收和发,保证不会死锁

集合通信

  • 集合通信: 涉及通信子中所有进程的通信
    所以进程都有调用集合通信的函数

  • 点对点通信: MPI_Send() MPI_Recv()

  • 点对点是通过标签和通信子匹配
    集合通信不使用表标签，只使用通信子和调用顺序匹配

  • 内存位置的名称与调用的匹配无关
