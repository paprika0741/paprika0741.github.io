操作系统原理(4)死锁

死锁的概念

死锁：一个进程集合中的每个进程都在等待只能由该集合中的其他一个进程才能引发的事件（释放占有资源/进行某项操作）
饥饿：由于长期得不到想要的资源，进程无法推进
死循环：某进程执行过程中一直跳不出某个循环的现象，程序的bug

共同点 区别 原因
死锁 进程无法顺利向前推进 1.一定是循环等待对方手里的资源，至少两个或两个以上的进程同时发生死锁
2.发生死锁进程一定再阻塞态		 系统资源的竞争和进程推进顺序不合理
饥饿 1.可能只有一个进程饥饿
2.发生饥饿进程可是阻塞态（如长期得不到需要的I/O设备），也可能是就绪态（长期得不到的处理机）		 系统分配资源不合理
死循环 死循环进程可以是允许态，只不过无法像期待那样顺利推进 （管理者）程序代码出问题

资源分类：

可重用资源:

  • 一次仅供一个进程使用，不因使用而耗尽
  • 使用后释放 能够被再次使用
  • 处理器 I/O通道 主存/辅存

可消费资源：

  • 可以创建(生产)并且可以销耗掉的资源
  • 数目没有限制，当一个进程得到一个可消费资源时，这个资源就不再存在了
  • 中断、信号、消息、I/O缓冲区中的信息

永久性资源
临时性资源

死锁产生的条件

互斥条件：

进程对所分配到的资源进行排他性使用
即在一段时间内某资源只由一个进程占有。

此时还有其他进程要求该资源，要求者只能阻塞，直至占有该资源的进程用毕释放。

占有且等待条件（ 请求和保持）：

进程已经占有至少一个资源，可以提出新的资源要求
若该资源已被其它进程占有，则请求进程阻塞，同时对已经获得的其它资源保持不放

不剥夺条件：

进程已获得的资源在未使用完之前不能被剥夺，只能在使用完时主动释放

前三个是必要条件而非充分条件

循环等待条件：

存在一种进程资源的循环等待链，链中每一个进程都有已获得的资源被下一个进程请求

如果同类资源大于1，即使循环等待，未必发生死锁。若系统中每类资源只有1个，循环等待是死锁的充要条件

​

死锁的处理策略

死锁预防 (deadlock prevention)

破坏死锁必要条件中的一个或者多个，保证不会发生死锁

破坏互斥条件

方法：不直接操作资源或多个进程同时使用虚拟资源
适用：

  • 资源的固有特性允许多个进程同时使用（如文件允许多个进程同时读）

  • 借助特殊技术允许多个进程同时使用（如打印机借助Spooling技术）

缺点：不适用于绝大多数设备

破坏占有且等待条件

方法：

  • 静态方法:
    进程在开始时一次性地申请在整个运行过程所需的全部资源并且保持占用,禁止已拥有资源的进程再申请其他资源
  • 申请资源时要先释放其占有资源后，再一次性申请所需全部资源

优点：
优点:简单、易于实现
缺点:

  • 进程运行期间一直占用,资源利用率低
  • 申请所有资源才能开始运行,进程延迟运行,还可能导致饥饿

破坏不剥夺条件

方法：

  • 一个已经占有了某些资源的进程，当它再提出新的资源请求而不能立即得到满足时，必须释放它已经占有的所有资源，待以后需要时再重新申请
    (会导致饥饿)
  • 当请求的资源被其他进程占用,操作系统协助剥夺该资源

适用: 资源的状态可以很容易地保存和恢复(如CPU)

缺点:

  • 实现复杂、代价大，
  • 反复申请/释放资源、系统开销大、降低系统吞吐量
  • 释放已获得资源可能会导致前一阶段工作失效,因此只适用于易于保存和恢复状态的资源,如CPU

破坏环路等待条件

方法:

  • 要求每个进程任何时刻只能占有一个资源，如果要申请第二个则必须先释放第一个（不现实）
  • 给资源编号,进程申请资源必须严格按资源序号递增的顺序,同类资源(序号相同的资源)一次性申请完
    (原因:已经占用小编号进程的才能申请大编号,持有大编号资源的进程不可能申请小编号,就不会循环等待)

缺点:

  • 增加新的设备需要重新编号
  • 很难找到令每个人都满意的编号次序，类型序号的安排只能考虑一般作业的情况, 限制了用户简单、自主地编程
  • 编号和实际使用顺序不一样,会不必要地拒绝对资源的访问 ,造成资源的浪费
  • 可能低效（会使进程的执行速度变慢）

死锁避免

  • 允许三个必要条件，通过明智的选择，确保不会到达死锁点
  • 相比死锁预防，允许更多的并发
  • 并不能确切预测死锁，仅是预料思索地可能性并确保不会出现这种可能性

死锁避免的实质：避免进入不安全状态

  • 如果一个新进程的资源请求会导致不安全状态， 则拒绝启动这个进程(进程启动拒绝)
  • 如果满足一个进程新提出的一项资源请求后，会导致不安全状态，则拒绝分配资源给这个进程（资源分配拒绝）

进程启动拒绝

  • 只有满足进程最大请求以及新的请求时，才会启动
  • 不是最优的，假设了最坏情况：所有进程同时发出最大的请求

  • 不能保证资源的最优使用率，但可以保证系统现有进程不会发生死锁

$R_j=V_j+\sum_{i=1} ^N A_{ij}$ ,所有资源，要么可用，要么已分配

$C_{ij} \leq Ri$ , 任意一个进程请求的资源不能超过这种资源总量

$A_{ij}\leq C_ {ij}$,分配的资源，不能超过最初数目的最大资源

若一个新进程的资源需求会导致死锁，则拒绝启动这个进程，当且仅当 $\forall j , \ \ \ R_j \geq C_{(n+1)j} + \sum_{i=1}^n C_{ij}$

资源请求拒绝/银行家算法

在资源动态分配过程中，某种方法防止系统进入不安全状态，从而避免死锁

安全状态：能够找到一个安全序列，为每个进程分配所需资源，直到满足每个进程对资源最大需求，使每个进程都可顺序完成。

不安全状态：无法找到一个安全序列，系统处于不安全状态

不安全状态 -> 可能死锁
死锁 -> 一定处于不安全状态
分配资源之前判断是否进入不安全状态

可用资源 Available
最大需求 Max
分配矩阵 Allocate
需求矩阵 Need
Need = Max- Allocate

优点：

  • 比死锁预防限制少

  • 无须死锁检测中的资源剥夺和进程重启

缺点：

  • 必须事先声明每个进程请求的最大资源

  • 考虑的进程必须是无关的，即它们执行的顺序没有任何同步要求的限制

  • 分配的资源数目必须是固定的

  • 在占有资源时进程不能退出

死锁的检查及解除

不限制资源访问或约束进程行为
只要有可能，就给进程分配请求的资源
周期性的执行一个算法检测循环检测条件

资源分配图

进程结点 对应一个进程
资源结点 对应一类资源，可能有多个
进程->资源 请求边，申请资源，每条边一个
资源->进程 分配边，分配资源，每条边一个

如果剩余资源不能满足进程需求：暂时阻塞
如果可以，则进程完成后将资源归还，删去边
如果最后能够消除所有的边，这个图是可完全简化的，此时一定没有发生死锁。

不能消除所有边，发生死锁

计算上相当于银行家算法，最后不能消去的就是死锁的

检测时机：

  • 资源申请时（早期检测，算法相对简单，但处理机时间消耗大）
  • 周期性检测，或死锁“好像”已经发生（如CPU使用率降低到一定阈值时）

检测方法：

  • 单个资源实例：检测资源分配图中是否存在环路（依据——死锁定理）

  • 多个资源实例：类似银行家算法的安全检查

消去所有的请求边与分配边 ..重复

死锁解除

资源剥夺法
  • 连续剥夺资源直到不再存在死锁
  • 挂起某些死锁进程，抢占资源，分配给其他的死锁进程
  • 注意被挂起进程长期得不到资源处于饥饿的状态
回退法

把每个死锁进程回滚(rollback)到前面定义的某些检查点(checkpoint), 并且重新启动所有进程（死锁可能重现）
要求系统保持进程的历史信息，设置还原点

撤销进程

杀死所有进程(最常用)
连续杀死死锁进程直到不再存在死锁
杀死一个非死锁进程

缺点: 杀死进程/剥夺资源 代价大
