#pragma

#pragma 默认长度一行，放不下 \ 转义字符

  • 特殊预处理器指令
  • 通常添加到系统中以允许不属于基本 C 规范的行为
  • 不支持编译指示的编译器会忽略它们
  • OpenMP的pragma以 #pragm omp 开始

pragm omp parallel …..

parallel指令

parallel指令表明后面的结构化代码块应该被多个线程并行执行

注意：:satisfied: 如果线程执行多条指令{}括起来，如果没有 {}，默认一条

最基本：

未说明线程数，运行时系统决定

​

pragm omp parallel num_thread(thread_count)

增加 num_thread 子句，子句只是用来修改指令的文本

  • 在parallel指令之前，程序只使用一个线程，
  • 到达parallel指令，原线程继续，另外thread_count - 1 个线程被启动
  • 不保证启动thread_count个线程
  • 程序可以启动的线程数可能存在系统定义的限制

  • 线程数过多 ./hello 10000

  • 顺序问题

因为对于标准输出的访问没有调度，
打印顺序是不确定的

  • 隐式路障

线程会等待所有的完成才会执行接下来的

临界区和锁

pragm omp critical

  • 表示对下面的代码块进行互斥访问

  • 一次只有一个线程执行下面的结构化代码

  • 如果没有命名，所有临界区一次只有一个线程访问——效率低

pragma omp critical(name)

不同名字的critical 保护的代码块可以同时执行
多个线程可以进入多个不同名字的临界区

但是 想让不同队列的线程同时访问相同的代码块， critical不能满足要求

atomic指令

pragma omp atomic

  • 只能保护一条C语言赋值语句临界区，效率更高
  • load-modify-store instruction

  • 必须是以下情形

    任意二元操作符
    expression不能引用x

只有x的装载和存储是可以确保保护的

pragma omp atomic

x++ = y++

y不受保护，x y 结果都不可测

lock 锁

显式强制互斥访问

  • 简单锁
    释放前只能获得一次

  • 嵌套锁
    释放前可以被同一个线程获得多次

比较三种临界区访问

atomic 最快 —所有atomic标志的强制互斥—相当于没有name的critical

可能左边执行右边不能执行，这种保护是不必要的

如果有多个atomic保护的临界区

则选择 name的critical 和 lock

lock 适用于互斥某个数据结构而不是代码块的情况

注意

  • 混合互斥
    critical和atomic不会互斥执行

    解决①右边修改g(x) 改成和x无关的显示，两边都是以atomic

      ②左边换成critical，两边都是critical

  • 互斥不保证公平性
    可能某个线程一直被阻塞

  • 嵌套互斥
    死锁

eg：梯形求和 见ppt 17

总之就是先串行算每一部分的，并行累加每一部分到结果

作用域

涉及parallel块中能访问该变量的线程集合

  • parallel块之前被声明的变量缺省是共享的
    线程组所有线程访问

  • 块中私有
    只能被单个线程访问

  • 被parallel for 指令并行化的块中，任何循环前声明的变量（循环变量除外）缺省情况都是 线程间共享的

  • default none字句，加上则编译器要求显示声明作用域

  • 规约变量同时私有和共享

归约子句 reduction

规约： 将相同的归约操作符 重复的应用到 操作序列来得到一个结果的计算
中间结果存储在归约变量里

归约变量：

reduction variable
所有操作的中间结果储存在归约变量中
同时有私有和共享作用域

归约操作符：

binary operation
减法可能有问题（不满足交换律和结合律）
浮点数也可能有问题（不满足结合律）

语法：

  • 变量本身是共享的（全局里定义）

  • 相当于创建局部私有变量，每个线程有一个局部的值计算暂存起来

  • 线程内涉及这个变量的语句都是处理私有变量
    （相当于共享的，每个线程有同名的私有的，全局的被遮蔽）

  • 然后创建一个临界区，将私有变量的值整合到共享变量里

    这里reduction(+: global_result) 理解局部变量，+默认初始值0

实际上每个thread 内部 0+= local_trap()

最后再和全局的初始值0 + 各个线程计算值加起来

注意 不同运算初始值不同，加法是0，乘法是1

reduction到底做了什么

线程结束计算是当前划分区域的值

所有线程结束后，才会全部计算在在全局的global_result里

parallel for

直接在for循环前放置一条指令

parallel for 后面结构块必须是for循环， 粗略的块划分

作用域：缺省作用域是私有的，　每个ｉ都有ｉ副本

隐形路障：所有线程完成循环才会继续

approx 规约变量 是必须的： 否则是一个普通的共享变量，将会是一个无保护的临界区

限制：

  • 只能并行化典型的for(index = start ; end ; incr )
    必须循环前确定，循环内部不能改变
  • index必须是int/pointer
  • start 、end 、incr 必须是兼容类型
    如果index为指针，incre必须是int
  • index只能被for语句中增量表达式修改，其他地方不行
  • parallel for 这句 里面 循环变量i作用域私有
    (否则i++会是无保护的临界区)

依赖性：

  • 无法并行化，迭代之间有依赖性 比如Fibonacci数列

依赖一个或多个前导结果

  • 对于共享变量的改变影响
    被parallel for 指令并行化的块中，任何循环前声明的变量（循环变量除外）缺省情况都是 线程间共享的

    比如这里的facor是共享的

由于factor是共享的，可能在其他线程里面改变了，那么每个线程需要有自己的factor副本，既是私有作用域

希望均匀

schedule

static

执行前，轮转的方式分配chunksize块个迭代给每个线程

schedule 就相当于缺省的调度，直接块划分

为了确保程序迭代采取块划分，每个for 之前加上 schedule(static, n/thread_count)

dynamic and guided

迭代在执行过程中分配给线程

dynamic 每个线程执行迭代块（里面chunksize个线程），执行完了，再请求另一个迭代块
chunksize缺省1

guided 只是迭代块越来越小

auto

编译时和运行时系统决定

runtime

没有chunksize

调度策略运行时检查环境变量OMP_SCHEDULE 决定

选择

系统开销 guided > dynamic > static

  • 如果每次迭代几乎相同工作量，默认最好

  • 如果随着循环进行，工作量递增/递减，小的chunksize的static比较好

  • 如果事先不能确定，runtime
